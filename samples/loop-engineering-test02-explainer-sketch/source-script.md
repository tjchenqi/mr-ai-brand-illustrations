本周除了 Fable 5 以外，还有一个开始集中传播的新词： “Loop Engineering”。Agent 的进展是日新月异，根本学不过来。但实际上关于 Loop Engineering 的技术结构早就出现了。22 年的 ReAct 把推理、行动与环境观察组织成连续迭代；2025 年的 Ralph 用外层循环反复启动 Coding Agent，并借助仓库文件、Git 提交和新上下文维持进度；Anthropic 的长任务实践也强调进度文件、跨上下文接力和端到端验证。Loop Engineering 的技术实践与名称传播双线时间轴因此，Loop Engineering 更像是对既有 Agent loop、harness、外部状态、自动验证和任务调度实践的重新组织，而不是突然出现的新技术。我和 GPT-5.5 讨论的在本文中采用工作定义如下：Loop Engineering 是围绕 Agent 构建可重复运行的目标—行动—反馈系统，并工程化设计其中的任务选择、外部状态、结果验证、预算、权限、重试、人工升级和退出条件。一、Agent 本来就是 Loop，为什么还要再一个 Loop？要理解这个定义，需要区分三个层次。第一层是 Agent loop，即单次 Agent 运行内部的循环：系统装配指令、目标和环境信息，模型产生文本或工具调用，运行时执行工具，把结果追加回上下文，再让模型决定继续还是结束。OpenAI 对 Codex agent loop 的拆解清楚展示了这一流程。第二层是 Agent harness，可以理解为承载 Agent loop 的运行系统。它负责选择和压缩上下文，提供工具与权限，准备沙箱或代码仓库，记录日志与状态，并限制时间、Token 和运行生命周期。Harness 不只是一个更长的 Prompt，它决定 Agent 在什么环境和约束下工作。第三层才是本文要说的 Loop Engineering。它关心的不只是这一轮 Agent 如何行动，而是为何要启动下一轮：现在有哪些工作？先做哪一项？结果是否真的满足目标？下一轮应该继续、重试、换策略、回滚、交给人，还是停止？可以把三者简化为：• Agent loop 负责这一轮如何行动；• Agent harness 负责这一轮在什么环境中行动；• Loop Engineering 负责整个任务为什么进入下一轮，以及何时结束。Agent loop、Agent harness 与 Loop Engineering 的三层关系（注意，这只是我这篇文章的分析模型。不同团队对 harness、runtime 和 orchestrator 的边界并不完全一致）。另外，在 Loop Engineering 中，Prompt Engineering 也没有消失。目标、上下文和局部 Prompt 影响每一轮，只是 Prompt 变成了 Loop 中的一个部件。而且，Loop Engineering 中 Workflow 与 Agent 也不必二选一。权限、预算和硬性验收可由确定性代码控制，开放式理解与生成再交给模型。Multi-agent 同样不是必要条件。二、一个 Loop 是怎样真正闭合的？假设团队希望自动处理一批低风险问题：新出现的 CI 失败、有明确复现步骤的缺陷，以及静态检查产生的机械任务。最容易想到的实现，是写一个循环：测试失败就调用 Coding Agent，Agent 修改代码后再跑测试，直到测试变绿。这已经有了循环的形状，却还没有形成可靠的闭环。因为每次进入下一轮之前，系统都必须回答几个问题：什么才算修好？这一轮应该处理哪个问题？Agent 可以做什么？结果由谁判断？失败之后又该怎么办？把这些问题展开，一个最小的任务级 Loop 才会出现：目标 → 读取状态 → 选择工作 → 执行 → 验证 → 写回 → 路由或停止任务级 Loop 的七个核心部件七步不是七个可以随意拼装的功能。前一步产生的信息，正是后一步作出决定的依据。运行前：先把意图变成一份验收契约系统收到的任务可能只是修复登录问题。这是一句意图，不是一个能驱动循环的目标。Agent 改了代码以后，系统仍然不知道该继续还是结束。因此，第一步不是调用模型，而是把目标改写成可以检查的结果，例如：“给定复现步骤不再返回 500；原有登录测试继续通过；不修改公开接口。”其中既有必须满足的硬约束，也有不能越过的边界。目标与验证在这里其实是一份契约：前者规定要证明什么，后者负责提供证据。目标与验证如何形成可执行的验收契约有了目标，系统还不能立刻开工。它需要读取上一轮留下的状态：哪些尝试已经失败，当前分支包含什么修改，哪些测试仍未通过，是否还有预算。上下文窗口适合当前推理，却不是可靠的长期数据库。跨轮次真正持续存在的，应当是任务清单、结构化进度、Git 提交、测试结果和未解决问题。状态决定了候选工作，选择器再从中挑出下一项值得执行的任务。一个目标含糊、无法验证或需要越权的任务，即使排在队首也不应启动。模型可以理解 Issue 和日志，确定性代码则更适合执行风险门槛、依赖检查和并发互斥。所谓“选择下一项”，不是简单地从队列取一个元素，而是根据当前状态重新判断：这一轮是否仍值得做？运行中：给 Agent 能力，也限定错误的作用范围选定任务后，系统才进入执行阶段。Coding Agent 需要代码库、终端和测试能力，但这些能力不应自动等于整个仓库、生产凭据和无限制的外部访问。更稳健的做法是，为这一轮创建隔离的 worktree 或沙箱，只注入相关上下文，并授予完成任务所需的最小权限。这也是为什么多 Agent 并行不是默认答案。只有任务能够清楚拆分、环境可以隔离、结果能够合并时，并行才可能提高吞吐；否则，它首先增加的是重复认领、文件冲突和状态覆盖。在这一阶段，模型负责开放式理解、规划和生成，普通代码负责权限、预算、环境和硬约束。所谓“确定性外壳约束非确定性内核”，不是要消除模型的不确定性，而是防止一次错误判断获得不受限制的执行能力。确定性控制外壳与非确定性模型内核运行后：先验证现实，再决定是否相信 Agent很多时候 Agent 只能说明它认为任务已经完成。Loop 要闭合，还必须回到环境中检查实际结果，比如复现步骤是否通过，原有测试是否回归，静态检查是否正常，修改范围是否越界。过程证据和结果证据也不能混为一谈。工具调用和代码 diff 说明 Agent 做了什么，测试、行为回放和任务专用 grader 才回答结果是否满足目标。生成者与验证者分离通常有助于减少自我确认，但验证者同样可能出错；测试通过也只证明被覆盖的属性成立。多轮行动、随机性和环境状态，使 Agent eval 尤其困难。[10]验证结果随后必须写回外部状态。系统不要保存对应的提交、diff、测试证据、失败类型和剩余工作。下一轮使用干净上下文时，读取的正是这份压缩后的现实。**干净上下文必须与持久状态一起使用。持久状态、干净上下文与结构化写回组成跨轮次循环进入下一轮：失败类型决定下一条路写回之后，Loop 才来到真正决定“是否循环”的位置。网络超时可以重试；实现错误可以携带测试证据进入新一轮；目标含糊应交给人；权限不足不能靠重复调用解决；触及高风险操作则应立即停止。因此，一个可运行的 Loop 不只有继续按钮，还必须有多个出口：目标满足、没有合适任务、预算耗尽、连续失败、风险触发或需要人工判断。轮次、时间、Token、外部 API 和变更范围也需要硬上限。没有退出条件的循环，不是自治，只是未受控运行。回到自动修复案例，一轮完整运行现在可以表示为：自动修复项目问题的一轮工作流while budget_available:
